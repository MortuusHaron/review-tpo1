ArcTg.java
в целом можно ограничить глубину рекурсии в 

### 1.2 Структура и проектирование
- соблюдение принципа единственной ответственности;
- декомпозицию;

- инкапсуляцию;

- смешение бизнес-логики и инфраструктуры.

В классе ThrowPersonAction жесткая зависимость от конкретного класса MCStudent. При добавлении новых типов студентов, которых можно выкинуть (если анекдот дополнится ещё занимательными персонажами) придется менять код (жесткие зависимости, хоть и в ограниченном контексте допустимо)
```
   @Override
    public void execute() {
        train.getPassengers().stream()
                .filter(p -> p instanceof MCStudent)
                .findFirst()
                .ifPresent(victim -> train.removePassenger(victim));
    }
```

Класс Train (removePassenger) изменяет статус студента (нарушение инкапсуляции), а из-за этого логика размазана по классам и при расширении функционала отслеживать логику программы станет сложнее

## ЧАСТЬ 5. Оценка через модель качества ISO/IEC 25010

### 5.1 Maintainability (сопровождаемость)
- **Modularity** — насколько код разделён на независимые компоненты?
- **Analyzability** — насколько легко понять логику и выявить дефекты?
- **Modifiability** — насколько легко внести изменения?
- **Reusability** — можно ли повторно использовать компоненты?
- **Testability** — насколько легко писать тесты для модулей?

- Modularity - хорошо (за исключением ThrowPersonAction)
- Analyzability - хорошо, есть сопровождающие описание что нужно сделать и как (в частности UML модель для понимая логики работы Domain), но есть часть с размазанной логикой (выход из поезда) что при расширении функционала может зааффектить
- Modifiability - нормальная, т.к. всё разделено на модули, но есть некоторые моменты как ThrowPersonAction, которые усложняют это
- Reusability - нормально
- Testability - нормально, т.к. есть требование с отслеживаем trace команды нужно понимать алгоритм работы модуля отвечающего за сортировку
